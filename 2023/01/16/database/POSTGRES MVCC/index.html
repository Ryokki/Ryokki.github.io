概述

并发控制是用于维护事务的一致性和隔离性，有三种并发控制策略：

  1. 多版本并发控制（Multi-Version Concurrency Control, MVCC）：维护数据的多个版本，通过获取数据版本快照进行事务隔离，使得每个快照版本都能拿到对应可见的数据版本。（这样就不需要通过行的读写锁来并发控制了，读和写可以并发进行）
  2. 严格两阶段锁（S2PL）：依赖锁来进行事务隔离
  3. 乐观的并发控制：在事务commit时才进行冲突判断（适合冲突概率小的情况）

MVCC是一种通过冗余多份历史数据来达到并发读写目的的一种技术，在写入或更新数据时，旧版本的历史数据将不会被删除，那么此时并发的读仍然能够读取到对应的历史数据，这样就使得读和写能够并发运行，并且不会出现数据不一致的问题。
在实现 MVCC 时，主要有两种方式:

  1. 将多版本的记录都存储在数据库中，垃圾收集器清理不需要的记录 (如Postgresql)
  2. 将最新的记录放在数据库中，旧数据放在undo log，其他线程在读取改行数据时，从回滚段中将旧数据读出来 (如Oracle, Mysql)

与 MVCC 相关的字段只有 4 个，其含义如下:

  • t_xmin: 保存了插入该元组的事务的 txid
  • t_xmax: 保存删除的txid 【更新 ～= 删除+插入】，若一个 tuple 既没有被更新也没有被删除的话，该字段的值为 0
  • t_cid: 即 Command ID，表示在当前事务中，执行当前命令之前共执行了多少条命令，从 0 开始计数。t_cid 的主要作用就在于判断游标的数据可见性，将在后文详细描述
  • t_infomask: 位掩码，主要保存了事务执行的状态，如 XMIN_COMMITTED、XMAX_COMMITTED 等。同时也保存了 COMBOCID 这一非常重要的标识位，也是和游标相关的字段。

对当前事务来说，每一个XMIN值比当前事务的XID要小（且xmax = 0 或 > 当前事务?）、并且该事务是已提交的元组，对于当前事务都是可见的。

具体:

  • 当事务ID小于xmin的事务表示已经被提交，其涉及的修改对当前快照可见
  • 事务ID大于或等于xmax的事务表示正在执行，其所做的修改对当前快照不可见
